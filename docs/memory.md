# Memory System

agent-persona gives Cursor AI agents **persistent memory across chat sessions**. This document explains how it works.

---

## The Problem

AI chat sessions are stateless. Every new chat starts from zero.

- Users repeat themselves: "Remember, I prefer tabs over spaces" — again and again
- Agents forget decisions: "We decided to use FastAPI for the backend" — lost in the next session
- Context evaporates: project conventions, architecture choices, personal preferences — all gone when the chat ends

agent-persona solves this by layering a memory system on top of the chat. The agent can recall past sessions, decisions, and preferences instead of starting fresh each time.

---

## Memory Layers

The system has two complementary layers: **episodic memory** (raw experiences) and **knowledge** (consolidated facts).

### Episodic Memory (`data/episodic/`)

Each session produces a **JSON snapshot** that captures:

- What was discussed
- Decisions made
- Technical context
- Emotional tone
- Time

These are the raw "experiences" — detailed but numerous. One episode per session (or per save point within a session).

**Storage:**

- Recent episodes live in `data/episodic/`
- Older ones are archived to `data/episodic/archived/`

**Written by:** the `prepare-handoff` and `store-episodic-memory` tasks.

**Example:** After a session where you refactored auth and chose JWT over sessions, an episode might capture that decision, the rationale, and the files touched.

---

### Knowledge (`data/knowledge/knowledge.json`)

Knowledge is **consolidated facts and patterns** extracted from many episodes:

- User preferences (communication style, tool preferences, coding conventions)
- Project conventions (architecture, patterns, naming)
- Technical decisions and their rationale
- Relationship context (how you work together)

Knowledge is more **compact and durable** than individual episodes. It's the distilled "what we know" rather than "what happened."

**Generated by:** the `infer-knowledge` task, typically at end-of-day when you signal you're done (e.g. "good night", "logging off").

**Queried by:** the `query-knowledge` task when the agent needs to recall something.

---

## Session Handoff (`data/current_session_handoff.md`)

The handoff file is a **markdown summary** of the most recent session. It acts as a bridge between chats: the next `conversation-start` reads it.

**Contains:**

- What was worked on
- What's pending
- Important context for the next session

**Design:** Lighter weight than full episodes — optimized for quick loading at the start of a new chat.

---

## Memory Lifecycle

How memory flows through the system:

1. **During a session:** `per-turn-check` monitors save timing and reminds the agent when a save is due.

2. **Periodically (or on user request):** `prepare-handoff` runs. It:
   - Saves an episodic memory snapshot
   - Updates the handoff file

   Triggers: user says "save", "remember", "prepare handoff", "switching", or after substantial work.

3. **End of day:** When you signal you're done ("good night", "logging off", "done for today"):
   - `prepare-handoff` runs (normal save)
   - `infer-knowledge` consolidates recent episodes into `knowledge.json`

4. **Over time:**
   - Old episodes move to `data/episodic/archived/`
   - Knowledge grows and refines as more episodes are processed

5. **Visualization:** The knowledge graph (`memory_graph.json`, `memory_graph.html`) shows connections between knowledge items — useful for understanding what the agent remembers and how it's structured.

---

## What Gets Remembered

Types of information the system captures:

| Type | Examples |
|------|----------|
| **Technical decisions** | "Using FastAPI for the backend", "PostgreSQL over MySQL" |
| **User preferences** | Communication style, tool preferences, coding conventions (tabs vs spaces, naming) |
| **Project context** | Architecture, patterns, conventions, key files |
| **Personal context** | Reminders, important dates mentioned |
| **Procedural notes** | User-approved rules for agent behavior |

---

## Retrieval

How memory is accessed:

- **conversation-start** loads eagerly at the beginning of each new chat:
  - Handoff file
  - Personality settings (base persona + active preset)
  - Procedural notes

  It does *not* load the full knowledge base. It only checks `knowledge.json` for `last_infer_date` to decide if consolidation is needed.

- **query-knowledge** retrieves knowledge on-demand when the agent needs to recall something specific (e.g. "What did we decide about auth?"). Semantic lookup against `knowledge.json` happens lazily, not at session start.

- The agent can **reference knowledge proactively** when relevant — for example, applying a stored preference without being asked.

## Summary

| Component | Purpose |
|-----------|---------|
| Episodic memory | Raw session snapshots; detailed, numerous |
| Knowledge | Consolidated facts; compact, durable |
| Session handoff | Quick bridge between chats |
| infer-knowledge | Episodes → knowledge (end-of-day) |
| query-knowledge | Semantic lookup when agent needs to recall |
